# Feature: Discount Codes

> Features: F-3.7  |  Status: DRAFT  |  Date: 2026-06-26
> PRD: DEC-1 (no platform fee, transparent pricing), QG-2 (transparency), QG-3 (fairness)
> DDD: BC-3 Sales, AGG-Order, VO-DiscountCode, SVC-DiscountPolicy, INV-20
> Tech: §4 (CQRS pipeline), §6 (persistence, concurrency)

## 1. Problem & Solution

**Problem:** Organizers have no way to offer promotional pricing — early-bird incentives, loyalty codes, or group discounts — without manually adjusting ticket prices, which affects all buyers and cannot be targeted or limited.

**Solution:** Let organizers create discount codes per event. Each code has a type (percentage or fixed amount), an optional validity window (start/end dates), and an optional usage cap. A valid code applied at checkout reduces the order total transparently — the discounted total is the final amount charged, consistent with EventHub's all-inclusive pricing principle.

**Personas:** PER-O1 (individual organizer) and PER-O2 (small group/club organizer) create and manage codes. PER-A1 and PER-A2 (attendees) apply codes during checkout.

**Scope:**
- **In:** Create/edit/delete discount codes per event; percentage and fixed-amount types; optional validity window; optional usage cap; apply code at checkout; transparent price reduction; owner-only management; reject invalid/expired/exhausted/unknown codes.
- **Out:** Bulk code generation, code sharing across events, attendee-facing code listing, analytics on code usage (future reporting via EP-9).

## 2. Acceptance Criteria

**AC-01:** GIVEN I hold the Owner role for an event, WHEN I create a discount code with a unique code string, a type (percentage or fixed amount), and a value, THEN the code is attached to the event and can be applied at checkout.

**AC-02:** GIVEN I hold the Owner role, WHEN I create a discount code with a percentage type and a value of 20, THEN applying this code reduces the order total by 20%.

**AC-03:** GIVEN I hold the Owner role, WHEN I create a discount code with a fixed-amount type and a value of 50, THEN applying this code reduces the order total by the equivalent of 50 (in the configured currency).

**AC-04:** GIVEN I hold the Owner role, WHEN I create a discount code with a validity window (start and end dates), THEN the code is only valid between those dates.

**AC-05:** GIVEN I hold the Owner role, WHEN I create a discount code with a usage cap of 10, THEN the code can be used at most 10 times across all orders for that event.

**AC-06:** GIVEN a valid, unexpired, non-exhausted discount code, WHEN an attendee applies it at checkout, THEN the order summary shows the reduction and the final total equals the discounted amount — the amount charged.

**AC-07:** GIVEN a discount code whose validity window has passed, WHEN an attendee tries to apply it, THEN the code is rejected with a clear message and the price is unchanged.

**AC-08:** GIVEN a discount code whose usage cap has been reached, WHEN an attendee tries to apply it, THEN the code is rejected with a clear message and the price is unchanged.

**AC-09:** GIVEN an unknown or misspelled code, WHEN an attendee tries to apply it, THEN the code is rejected with a clear message and the price is unchanged.

**AC-10:** GIVEN a percentage discount code that would reduce the total below zero (e.g., 100% off a line), WHEN it is applied, THEN the final total is clamped at zero (never negative) per INV-20.

**AC-11:** GIVEN I hold the Owner role, WHEN I edit a discount code's value, validity window, or usage cap, THEN the changes apply to future uses only — existing orders that already used the code are unaffected.

**AC-12:** GIVEN I hold the Owner role, WHEN I delete a discount code, THEN it can no longer be applied to new orders; existing orders that used it retain their original discounted total.

**AC-13:** GIVEN I do not hold the Owner role for an event, WHEN I attempt to create, edit, or delete a discount code for that event, THEN I am refused with an "insufficient permissions" message.

**AC-14:** GIVEN an attendee applies a discount code at checkout, WHEN the order is placed, THEN the code and its reduction are recorded on the order for transparency and audit (the order line items and total reflect the discount).

## 3. Domain & Business Rules

**Bounded context:** Sales (BC-3) — discount codes are part of the purchase flow, not the event/ticket-type definition in Event Management.

**Value object — Discount Code:**
- A code has: a unique string identifier (per event), a type (percentage or fixed amount), a value, an optional validity window (start/end), an optional usage cap, and a usage counter.
- Percentage values are between 1 and 100 inclusive.
- Fixed-amount values are positive and in the event's configured currency.
- The code string is case-insensitive for attendee input but stored in a normalized form.
- A code is unique within an event (the same string may exist for different events).

**Domain service — Discount Policy:**
- Validates a code: checks existence, type/value, validity window, usage cap.
- Computes the reduction: percentage → `line total × (value / 100)`; fixed → subtract from total, clamped at zero.
- Returns the discounted total; does not mutate the order directly.

**Aggregate — Order:**
- An order may reference zero or one applied discount code.
- INV-20: `Total = Σ line totals − discount`, and `Total ≥ 0`.
- When a code is applied, the order records the code identifier and the computed reduction at placement time — the discount is snapshot, not recalculated later.
- Once an order is placed, its discount is immutable (editing or deleting the code does not retroactively change confirmed orders).

**Invariants:**
- A discount cannot make the order total negative (INV-20 — total ≥ 0).
- A code's usage counter increments atomically when an order using it is placed, preventing oversubscription under concurrent checkouts.
- A code can only be applied to orders for the event it belongs to.

**Domain events:** No new domain events are required. The discount is part of `EVT-OrderPlaced` (the order total already reflects it). Code management (create/edit/delete) is a configuration action, not a domain event.

## 4. UI Behavior

### Organizer — Manage discount codes

- Accessible from the event management area (alongside ticket types).
- List shows: code string, type, value, validity window (if set), usage (used / cap, or "unlimited"), status (active / expired / exhausted).
- Create form: code string (text), type (percentage or fixed amount toggle), value (number), validity window (optional start/end date-time pickers), usage cap (optional number).
- Edit: same form; changes affect future uses only (a confirmation note states this).
- Delete: confirmation dialog; warns that existing orders are unaffected.

### Attendee — Apply at checkout

- A "Discount code" input field on the checkout summary.
- On entry and submit: validate against the API; show the reduction and updated total on success, or an inline error on failure.
- The discounted total is the final amount — no additional fees (consistent with DEC-1 and QG-2).
- Only one code per order (simplicity — QG-1).

## 5. Data & Storage Impact

**PostgreSQL (app schema):**
- New table for discount codes: id, event_id (FK to events), code (normalized), type (percentage or fixed), value, start_at (nullable), end_at (nullable), usage_cap (nullable), used_count (default 0), created_at, updated_at, row_version.
- Unique constraint on (event_id, code) — case-insensitive via normalized storage.
- Orders table: add nullable discount_code_id FK and discount_amount column (snapshot of the reduction applied).

**Redis:** No new cache requirements. Discount codes are low-frequency management operations; direct PostgreSQL queries are sufficient.

**RabbitMQ:** No new integration events. Discount application is within the Sales context boundary.

## 6. Real-Time & Consistency

**Concurrency:** When multiple attendees apply the same capped code simultaneously, the usage counter must be updated atomically (optimistic concurrency with retry, consistent with the project's existing approach for inventory in Event Management).

**Consistency:** Discount validation and application happen within the PlaceOrder transaction (strong consistency). The usage counter increment is part of the same unit of work as order placement — no partial state.

**Real-time:** No SignalR impact. Discount management is an organizer configuration action, not a live-updating concern.

## 7. Security & Privacy

- Only organizers with the Owner role can manage discount codes (enforced in Application handlers via `ICurrentUserAccessor`, consistent with F-1.5).
- Discount codes are per-event and cannot leak across events.
- No sensitive data in discount codes — they are promotional strings, not credentials.
- The attendee-facing API for applying codes should not reveal whether a code exists for a different event (generic "invalid code" error).

## 8. Edge Cases

**EC-01:** A percentage code with value 100 reduces the total to zero. This is valid and results in a free order — auto-confirmed per F-6.2 (no payment step required).

**EC-02:** A fixed-amount code with a value greater than the order total. The total is clamped at zero (never negative per INV-20). The order is treated as free.

**EC-03:** An attendee applies a code, then removes it before placing the order. The code's usage counter is not incremented — only successful order placement counts.

**EC-04:** An attendee applies a code, then the organizer deletes the code before the order is placed. The checkout should re-validate on placement; if the code no longer exists, the order proceeds at full price with a notification to the attendee.

**EC-05:** Two attendees apply the last use of a capped code at the same instant. Optimistic concurrency ensures exactly one succeeds; the other sees a "code exhausted" error and pays full price.

**EC-06:** A code's validity window ends while an attendee is mid-checkout (code was valid when they started). Re-validation on placement rejects the code; the attendee sees an error and can proceed at full price.

**EC-07:** The same code string is used for different events. This is allowed — codes are scoped per event.

**EC-08:** An organizer creates a code, applies it to their own test order, then deletes the code. The test order retains its discounted total (snapshot principle).

## 9. Dependencies & Risks

**Dependencies:**
- F-3.1 (Define a ticket type) — discount codes attach to events that have ticket types.
- F-3.3 (Transparent, all-inclusive pricing) — the discounted total must be the final amount; no hidden fees.
- F-5.3 (Create order and hold inventory) — codes are applied during order placement.
- F-6.2 (Free-order auto-confirm) — a fully discounted order (total = 0) triggers immediate confirmation.

**Risks:**
- **Atomic usage counter increment** — if the increment is not part of the order transaction, concurrent checkouts could oversubscribe a capped code. Mitigation: include the counter update in the PlaceOrder unit of work with optimistic concurrency.
- **Code validation timing** — codes must be re-validated at placement time, not just at checkout entry, to handle expiry/deletion during the attendee's session. Mitigation: validate in the PlaceOrder command handler.

## 10. Assumptions

- One discount code per order (simplicity; QG-1). Stacking or combining codes is out of scope.
- Codes are managed per-event, not globally. Cross-event codes add complexity without clear value at this scale.
- The discount is applied to the order total, not to individual line items. Per-line-item targeting is a future enhancement if needed.
- No attendee-facing directory of available codes — codes are shared by the organizer through their own channels (email, social, etc.).

## 11. Out of Scope

- Bulk code generation (e.g., generate 100 unique single-use codes at once).
- Automatic codes (e.g., "early-bird" pricing that activates/deactivates by time — that is F-3.8, Scheduled on-sale window).
- Per-ticket-type discount targeting.
- Code analytics / usage reporting (deferred to EP-9).
- Attendee-facing code listing or discovery.
- Stacking multiple codes on one order.
