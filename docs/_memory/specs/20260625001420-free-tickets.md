# Feature: Free tickets

> Features: F-3.2  |  Status: DRAFT  |  Date: 2026-06-25
> PRD: DEC-1 (no platform fee), DEC-3 (MVP scope), QG-1 (simplicity), QG-2 (transparent pricing), QG-5 (no oversell)  |  DDD: BC-2 AGG-Event, BC-3 AGG-Order, ENT-TicketType, INV-13, INV-23  |  Tech: §4, §6, §7

## 1. Problem & Solution

**Problem:** Many small events — meetups, community talks, free workshops — charge nothing. If the system requires a payment step even for zero-price tickets, it creates unnecessary friction for both organizers (who just want to share an event) and attendees (who should be able to get a free ticket instantly). The ticket type feature (F-3.1) already allows a price of zero, but the checkout and payment flow has no special handling for free orders.

**Solution:** When an order's total is zero (all line items are free ticket types), the system skips the payment step entirely and confirms the order immediately. The attendee receives their tickets without any payment provider interaction. This applies to fully free events (all ticket types are zero-priced) as well as mixed scenarios where discount codes (F-3.7, future) reduce a paid order to zero.

**Personas:** PER-O1 and PER-O2 (organizers who run free events) and PER-A1 (attendees who want a frictionless free ticket).

**Scope:** F-3.2 only. The companion payment-side behavior (F-6.2 — free-order auto-confirm) is the downstream feature that implements the auto-confirmation triggered by a zero total. This spec covers the product-level behavior and the domain rules that make free tickets work end-to-end.

## 2. Acceptance Criteria

**AC-01:** GIVEN an event where all ticket types have a price of zero, WHEN an attendee selects tickets and proceeds to checkout, THEN no payment step is presented and the order is confirmed immediately.

**AC-02:** GIVEN a free ticket type (price = zero), WHEN an attendee checks out, THEN the order total is zero and the order transitions directly from Pending to Confirmed without any payment provider interaction.

**AC-03:** GIVEN a confirmed free order, WHEN the confirmation completes, THEN tickets are issued (F-7.1) and delivered to the attendee (F-7.2) exactly as they would be for a paid order.

**AC-04:** GIVEN an event with a mix of free and paid ticket types, WHEN an attendee selects only free ticket types, THEN the order total is zero and the free-order flow applies (no payment step).

**AC-05:** GIVEN an event with only free ticket types, WHEN the event is published (F-2.4), THEN the event page shows the ticket price as "Free" (or equivalent) and no price amount is displayed.

**AC-06:** GIVEN a free event, WHEN an attendee views the order summary (F-5.4), THEN the total is shown as zero (or "Free") with no hidden fees or charges.

**AC-07:** GIVEN a free ticket type, WHEN the ticket type is created (F-3.1), THEN the price of zero is accepted as valid (INV-13: price ≥ 0) and the ticket type is treated identically to a paid ticket type in all respects except payment.

**AC-08:** GIVEN a free order that has been confirmed, WHEN the organizer views the order, THEN it shows as Confirmed with a total of zero, and there is no associated payment record (since no payment was initiated).

## 3. Domain & Business Rules

Referenced from `domain-model-specification.md`:

- **INV-13 — Price ≥ 0:** A ticket type's price must be non-negative. Zero is a valid price representing a free ticket. This invariant is already enforced by F-3.1; F-3.2 relies on it.
- **INV-23 — Confirmation guard:** Confirmation requires a captured payment **or** a zero total (free). This is the key invariant that enables free-order auto-confirm — an order with `Total = 0` satisfies the confirmation guard without any payment.
- **VO-Money:** The money value object supports zero amounts. A total of zero is a valid `Money` value, not a special case or null.
- **AGG-Order behavior — `Place`:** When a free order is placed, the order is created in Pending status with `Total = 0`. The `Place` behavior does not distinguish between free and paid orders at this stage — the distinction happens at confirmation.
- **AGG-Order behavior — `MarkConfirmed`:** For a zero-total order, `MarkConfirmed` can be called immediately after `Place` without requiring a `Payment` to exist. The guard (INV-23) checks: is there a captured payment, OR is the total zero? If either is true, confirmation proceeds.
- **No Payment aggregate for free orders:** A `Payment` record (BC-4) is only created when a non-zero order needs to be settled through the external provider. Free orders skip Payment creation entirely — there is nothing to pay.
- **Inventory still applies:** Free ticket types still consume inventory (F-3.4). Reserving, committing, and releasing inventory works identically for free and paid ticket types. A free event can still sell out.
- **Ownership enforcement:** Only the event owner can define ticket types (F-3.1). This is unchanged — the price being zero does not alter authorization.

## 4. API Contract

**No new endpoint is required for F-3.2.** Free tickets are a natural consequence of the existing ticket type creation (F-3.1) and checkout (EP-5) flows. The behavior differences manifest in the existing endpoints:

- **Ticket type creation** (`POST /api/events/{eventId}/ticket-types`): A price of zero is already valid per F-3.1. No change.
- **Order placement** (`POST /api/orders`): The handler checks the order total after calculating line items. If total = 0, it skips payment initiation and calls `MarkConfirmed` immediately.
- **Order status** (`GET /api/orders/{orderId}`): A confirmed free order shows status `Confirmed` with total `0` and no payment reference.

**API response shape for a free order:**

| Field | Value |
|-------|-------|
| status | `confirmed` |
| total | `{ amount: 0, currency: "VND" }` |
| paymentId | `null` (no payment was created) |
| confirmedAt | timestamp of immediate confirmation |

## 5. Data & Storage Impact

- **PostgreSQL:**
  - **Orders table:** A free order is stored identically to a paid order, except `payment_id` is `null` and `confirmed_at` is set immediately (not after a payment webhook).
  - **Ticket types table:** No schema change — `price_amount = 0` is already a valid value.
  - **Payments table:** No row is created for free orders. This is a deliberate absence, not a sentinel value.
- **Redis:** No direct cache impact. The session and event caches are unaffected by whether a ticket type is free or paid.
- **MinIO:** No change.
- **RabbitMQ:** The `EVT-OrderConfirmed` integration event is emitted for free orders exactly as it is for paid orders. Downstream consumers (Ticketing for issuance, Notifications for delivery, Reporting) do not need to know whether the order was free or paid.

## 6. Real-Time & Consistency

- **Strong consistency:** Free-order confirmation is a single-transaction operation: place the order + mark confirmed + commit inventory reservation. This happens synchronously within the request, unlike paid orders which wait for an asynchronous payment webhook.
- **No race condition on "free vs paid" determination:** The order total is calculated server-side from the ticket type prices at the time of placement (INV-25 — price snapshotting). A ticket type cannot change from free to paid (or vice versa) between the attendee selecting it and the order being placed, because prices are snapshotted.
- **Inventory consistency:** Free orders still go through the reservation → commit flow (F-3.4 / F-5.3). Two attendees racing for the last free ticket are handled by the same no-oversell guarantee (INV-10) as paid tickets.

## 7. Security & Privacy

- **No payment data involved:** Free orders never interact with the payment provider. No payment session, no redirect, no webhook. This simplifies the security surface — there is no payment data to protect.
- **Guest checkout still applies:** Free tickets follow the same guest checkout flow (F-5.2). An attendee provides name and email; no account is required.
- **Rate limiting consideration:** Free events could attract higher volume. The existing inventory cap (capacity) serves as a natural rate limiter. Additional rate limiting on order creation is a future concern, not an MVP requirement.
- **Anti-abuse:** Free tickets are still subject to per-order purchase limits (F-3.6, when implemented) and inventory caps. There is no mechanism for an attendee to "hoard" free tickets beyond what the organizer allows.

## 8. Edge Cases

**EC-01:** An event has one free ticket type and one paid ticket type. An attendee selects one of each. The order total is non-zero (the paid ticket's price), so the normal payment flow applies. The free ticket type contributes zero to the total but still consumes inventory.

**EC-02:** An event has only free ticket types. An attendee selects multiple tickets. The order total is zero, the order is confirmed immediately, and multiple tickets are issued. The attendee receives all tickets by email.

**EC-03:** A free event sells out (all free ticket capacity is exhausted). Additional attendees see "sold out" and cannot place an order. Free does not mean unlimited.

**EC-04:** An attendee places a free order but the inventory reservation fails (last ticket was just taken by another attendee). The order is not created and the attendee sees a sold-out message. The free-order flow does not bypass the no-oversell guarantee.

**EC-05:** An organizer publishes an event with free tickets, then later changes a ticket type's price from zero to a non-zero amount (F-2.3 / F-3.5). Existing confirmed free orders are unaffected (prices are snapshotted per INV-25). New orders use the updated price.

**EC-06:** A discount code (F-3.7, future) reduces a paid order's total to zero. The same free-order auto-confirm logic applies — the order is confirmed immediately without a payment step. This is handled by the same INV-23 guard.

**EC-07:** An attendee starts checkout for a free event, reaches the order summary, but does not complete. The order remains Pending and the hold expires normally (F-5.5), releasing the reserved free tickets back to availability.

## 9. Dependencies & Risks

**Dependencies:**
- F-3.1 (Define a ticket type) — must exist; F-3.2 relies on price = 0 being a valid ticket type
- F-5.3 (Create order and hold inventory) — the order placement flow must exist for the free-order shortcut to work
- F-6.2 (Free-order auto-confirm) — the companion feature on the payment side that implements the immediate confirmation for zero-total orders
- F-7.1 / F-7.2 (Ticket issuance and delivery) — confirmed free orders produce tickets just like paid orders

**Risks:**
- **Misunderstanding "free":** Users may expect free events to have no capacity limits. The spec is clear that inventory still applies, but the UI (F-4.1, F-4.2) must communicate this clearly.
- **Abuse of free events:** Without payment as a barrier, free events could attract spam registrations. This is an operational concern, not a domain problem — inventory caps and per-order limits provide the necessary guardrails.
- **Future discount codes (F-3.7):** When discount codes reduce a paid order to zero, the same auto-confirm path must be used. This creates a coupling between F-3.2 and F-3.7 that should be designed for now (the INV-23 guard handles it cleanly).

## 10. Assumptions

- A ticket type with price = 0 is the only way to create a free ticket. There is no separate "free event" flag — an event is free if and only if all its ticket types have a price of zero.
- Free orders follow the same guest checkout flow as paid orders (F-5.2). There is no simplified "just give me a ticket" flow for free events — the attendee still provides name and email.
- The `EVT-OrderConfirmed` integration event is emitted for free orders, ensuring downstream consumers (ticket issuance, notifications, reporting) work identically regardless of payment status.
- Free orders do not create a `Payment` aggregate. The absence of a payment record is the indicator that the order was free, not a status field on the payment.
- The public event page (F-4.1) displays "Free" instead of a price amount when all ticket types are zero-priced. This is a UI concern handled by the frontend, not a domain rule.

## 11. Out of Scope

- **Discount codes (F-3.7):** Reducing a paid order to zero via a discount code. The domain guard (INV-23) supports this, but the discount code feature itself is specified separately.
- **Payment flow for paid orders (F-6.1):** The normal payment provider interaction for non-zero orders.
- **Per-order purchase limits (F-3.6):** Capping how many free tickets one order can claim.
- **Transparent pricing display (F-3.3):** How prices (including "Free") are shown on the public page and checkout summary.
- **Inventory and no-oversell (F-3.4):** The reservation and sold-tracking mechanism, which applies equally to free and paid tickets.
- **Attendee account linking (F-1.4):** Free tickets are delivered to guests; optional account linking is a Later feature.
