# Feature: Duplicate an Event

> Features: F-2.6  |  Status: DRAFT  |  Date: 2026-06-23
> PRD: QG-1 (simplicity), DEC-3 (MVP scope)  |  DDD: BC-2 AGG-Event  |  Tech: §4, §6

## 1. Problem & Solution

**Problem:** An organizer who has run an event before (e.g., a recurring workshop, monthly meetup, or seasonal class) must manually re-enter all event details and re-create every ticket type from scratch when setting up the next occurrence. This is tedious, error-prone, and discourages reuse of well-structured event setups.

**Solution:** A single "Duplicate" action on an existing event that creates a new Draft copy. The new event inherits the source event's descriptive details (title, description, location, cover image) and ticket-type definitions (names, prices, capacities), but starts fresh with no sales, no attendees, no dates, and no published status. The duplicating organizer becomes the Owner of the new event.

**Personas:** PER-O1 (individual organizer), PER-O2 (small group organizer).

**Scope:** F-2.6 only. The new event is a Draft — publishing (F-2.4) and all downstream flows are separate actions on the new event.

## 2. Acceptance Criteria

**AC-01:** GIVEN I hold the Owner role for a source event, WHEN I duplicate the event, THEN a new event is created in Draft status AND I am the Owner of the new event.

**AC-02:** GIVEN I am duplicating an event, WHEN the new Draft is created, THEN the following are copied from the source: description, location, cover image reference, and all ticket-type definitions (name, price, capacity). The title is copied with a "Copy of " prefix so the new Draft is distinguishable in the organizer's event list.

**AC-03:** GIVEN I am duplicating an event, WHEN the new Draft is created, THEN the following are NOT copied: schedule (start/end dates), sales data (sold counts, reserved counts), attendees, orders, tickets, check-in status, slug, and published/closed/cancelled status.

**AC-04:** GIVEN I am duplicating an event, WHEN the new Draft is created, THEN each copied ticket type starts with zero sold and zero reserved — it is a fresh inventory.

**AC-05:** GIVEN I do not hold the Owner role for the source event, WHEN I attempt to duplicate it, THEN I am refused with an "insufficient permissions" message and no new event is created.

**AC-06:** GIVEN a Draft, Published, Closed, or Cancelled event, WHEN the Owner duplicates it, THEN the action succeeds — any event in any status can be duplicated (the source is read-only for this operation).

**AC-07:** GIVEN I duplicate an event, WHEN the new Draft is created, THEN it has no slug (slugs are generated on publish per F-2.4) and no public URL yet.

**AC-08:** GIVEN I duplicate an event, WHEN the new Draft is created, THEN the new event's schedule is unset (the organizer must set new dates before publishing).

## 3. Domain & Business Rules

**What is copied (from domain-model-specification.md §4 BC-2 AGG-Event):**

| Source field | Copied? | Reason |
|-------------|---------|--------|
| Title | Yes (with "Copy of " prefix) | Reuse event identity; prefix distinguishes the new Draft |
| Description | Yes | Reuse event content |
| Location (physical or online) | Yes | Reuse venue/setup |
| Cover image reference | Yes | Reuse branding |
| Ticket-type definitions (name, price, capacity) | Yes | Reuse pricing structure |
| Schedule (start/end + time zone) | **No** | New occurrence needs new dates |
| Status | **No** | Always starts as Draft |
| Slug | **No** | Generated fresh on publish (INV-15) |
| Sold / Reserved counts | **No** | Fresh inventory (INV-10) |
| Reservations | **No** | No active holds on a new event |
| Orders, attendees, tickets | **No** | Belong to the source event's lifecycle |

**Authorization:**
- Only the **Owner** of the source event can duplicate it. This aligns with F-1.5 (Owner = full control).
- The duplicating user becomes the Owner of the new event regardless of any other roles they hold.

**Domain behavior:**
- `Event.Duplicate(organizerId)` — a factory-style behavior that reads the source aggregate's data and produces a new `Event` in Draft state with fresh identity.
- The new event gets a new `VO-EventId`. It is a completely independent aggregate — no link back to the source is maintained.
- Ticket types are deep-copied: each new `ENT-TicketType` gets a new `VO-TicketTypeId` and starts with `Sold = 0`, `Reserved = 0`.

**No domain events:**
- Duplicating is a private organizer action with no external side effects. No integration events are emitted. The event exists only as a Draft until the organizer explicitly publishes it.

## 4. API Contract

**Endpoint:** `POST /api/events/{eventId}/duplicate`

**Auth:** Cookie session required. The handler verifies the caller holds the Owner role for the source event.

**Success (201):** Returns the newly created Draft event with a new event ID. The response includes the copied details and ticket-type definitions but no schedule, no slug, and zero inventory counts.

**Error responses:**

| Status | Code | Condition |
|--------|------|-----------|
| 401 | `UNAUTHORIZED` | No active session |
| 403 | `FORBIDDEN` | Caller does not hold the Owner role for the source event |
| 404 | `NOT_FOUND` | Source event does not exist |

## 5. Data & Storage Impact

**PostgreSQL:**
- A new row is inserted into the `events` table with a new ID, status `Draft`, and copied descriptive fields. The schedule columns are null/unset.
- New rows are inserted into the `ticket_types` table for each copied ticket type, linked to the new event ID, with `sold = 0` and `reserved = 0`.
- The `row_version` on the new event starts at its initial value.
- The source event is read-only for this operation — no changes to its row.

**Redis / MinIO / RabbitMQ:**
- No cache invalidation needed — the new Draft is not publicly visible.
- The cover image reference is copied as a key/URL; the binary in MinIO is shared (no duplicate upload).
- No messages published to RabbitMQ.

## 6. Real-Time & Consistency

- **Strong consistency:** The duplicate operation creates the new event and its ticket types within a single transaction. There is no cross-aggregate write — the source event is only read.
- **No real-time push:** The new event is a private Draft. No SignalR notification is needed. Real-time updates (EP-11) apply only to published events with active sales.
- **No integration events:** Duplicating has no downstream consumers.

## 7. Security & Privacy

- **Session auth required:** The caller must be signed in. No guest or anonymous duplication.
- **Owner-only authorization:** Checked in the Application handler via `ICurrentUserAccessor` — not just at the API endpoint level.
- **No sensitive data exposed:** The source event's sales data, attendee lists, and orders are never included in the duplication. The new event starts clean.
- **Cover image sharing:** The new event references the same MinIO object as the source. If the source event's image is later deleted, the new event's reference becomes stale. This is acceptable — the organizer can re-upload.

## 8. Edge Cases

**EC-01:** Organizer duplicates a Draft event that has no ticket types → the new Draft also has no ticket types. The organizer must add ticket types before publishing (per F-2.4 / INV-11). This is valid — duplication preserves the source state.

**EC-02:** Organizer duplicates a Cancelled event → succeeds. The new event is an independent Draft. The source's cancelled status is not copied.

**EC-03:** Organizer duplicates an event they do not own (e.g., they hold Staff role only) → refused with "insufficient permissions."

**EC-04:** Source event's cover image has been deleted from MinIO between the organizer viewing the source and clicking Duplicate → the new event references a stale image key. The organizer can upload a new image on the Draft. The duplication itself succeeds.

**EC-05:** Source event has a ticket type with a sales window (F-3.8) → the sales window is copied as-is. The organizer should review and update it for the new event's dates before publishing.

**EC-06:** Organizer duplicates an event simultaneously from two browser tabs → two independent Draft events are created. Both are valid; the organizer can delete or edit either.

**EC-07:** Source event has a zero-capacity ticket type → copied as-is. Zero-capacity ticket types are valid (show as sold out immediately per EC-03 in the publish spec).

## 9. Dependencies & Risks

**Dependencies (already built):**
- F-1.2 (Sign in) — organizer must be authenticated.
- F-1.5 / F-1.6 (Roles) — Owner role must exist and be assignable.
- F-2.1 (Create draft event) — the new event is created as a Draft; the creation mechanism already exists.

**Dependencies (for full value):**
- F-3.1 (Define ticket types) — ticket types must exist on the source to be copied. If the ticketing epic is not yet built, duplication copies only event details with no ticket types.
- F-2.2 (Cover image) — copied by reference if present.

**Risks:**
- **Premature feature timing:** F-2.6 is in the "Next" phase. If built before F-3.1 (ticket types), duplication copies only event details — still useful but less valuable. Risk is low; the feature degrades gracefully.
- **Stale image references:** Cover images are shared by reference. Deleting the source event or its image does not affect the new event's stored key, but the binary may be orphaned or missing. This is a storage hygiene concern, not a correctness issue.

## 10. Assumptions

- The Event aggregate's `Duplicate` behavior can read the source aggregate's fields and construct a new aggregate. This is a read of the source and a write of the new event within one transaction.
- The new event has no back-reference to the source event. They are independent aggregates after duplication.
- Ticket-type IDs are regenerated for the new event — they are not shared with the source.
- The duplicate action is available in any event status (Draft, Published, Closed, Cancelled). The source is treated as a template.
- The cover image binary in MinIO is not duplicated — only the reference key is copied. This avoids unnecessary storage costs.

## 11. Out of Scope

- **Link between source and duplicate:** No "duplicated from" reference is maintained. The new event is fully independent.
- **Selective field copying:** The organizer cannot choose which fields to copy during duplication. All eligible fields are copied; the organizer edits the new Draft afterward.
- **Bulk duplication:** One event at a time.
- **Template system:** Duplicating is a one-off clone, not a managed template. A template library is a separate concept.
- **Auto-set dates:** The system does not guess or auto-fill dates for the new event. The organizer sets them manually.
- **Re-upload cover image:** The new event shares the source's image binary. A separate re-upload is the organizer's choice on the Draft.
