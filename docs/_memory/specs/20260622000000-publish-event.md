# Feature: Publish an event

> Features: F-2.4  |  Status: DRAFT  |  Date: 2026-06-22
> PRD: DEC-3 (MVP scope), QG-1 (simplicity), QG-5 (correctness)  |  DDD: BC-2 AGG-Event, INV-11, INV-15  |  Tech: §4, §6, §7

## 1. Problem & Solution

**Problem:** An organizer has created a draft event (F-2.1) and defined at least one ticket type (F-3.1), but the event is still private and invisible to potential attendees. There is no way for people to discover or buy tickets until the event is made public.

**Solution:** Provide a single action that transitions a draft event to published status. Publishing validates that the event is ready (has required details and at least one ticket type), generates a stable public slug/URL, and makes the event visible for discovery (EP-4) and purchasable (EP-5). The action is restricted to the event's Owner.

**Personas:** PER-O1 (individual organizer) and PER-O2 (small group organizer) — the people who own and manage events.

**Scope:** F-2.4 only. The public event page (F-4.1) and purchase flow (EP-5) consume the published state but are specified separately.

## 2. Acceptance Criteria

**AC-01:** GIVEN I hold the Owner role for a Draft event AND the event has all required details (title, start/end date-time with time zone, location) AND at least one ticket type, WHEN I publish the event, THEN the event status changes from Draft to Published AND a stable public URL is generated for the event.

**AC-02:** GIVEN I hold the Owner role for a Draft event AND the event is missing required details (e.g., no title, no schedule, no location), WHEN I attempt to publish, THEN publishing is blocked AND I receive a message listing each missing requirement.

**AC-03:** GIVEN I hold the Owner role for a Draft event AND the event has zero ticket types, WHEN I attempt to publish, THEN publishing is blocked AND I receive a message stating that at least one ticket type is required.

**AC-04:** GIVEN I do not hold the Owner role for the event, WHEN I attempt to publish, THEN I am refused with an "insufficient permissions" message and the event remains in Draft.

**AC-05:** GIVEN a Published event, WHEN I attempt to publish it again, THEN the action is rejected with a clear message that the event is already published.

**AC-06:** GIVEN a Closed or Cancelled event, WHEN I attempt to publish, THEN the action is rejected with a clear message indicating the event's current status does not allow publishing.

**AC-07:** GIVEN a Draft event that meets all requirements, WHEN it is published, THEN the generated slug is unique among all published events and is URL-safe.

**AC-08:** GIVEN a Draft event that meets all requirements, WHEN it is published, THEN an integration event is emitted so that downstream systems (reporting, discovery) are notified.

## 3. Domain & Business Rules

Referenced from `domain-model-specification.md`:

- **INV-11 — Publishable guard:** An event may only move from Draft to Published when it has all required details AND at least one ticket type attached. Attempting to publish without meeting these conditions is rejected with specific reasons.
- **INV-15 — Slug uniqueness:** Upon publishing, a URL-safe slug is generated that is unique among all published events. This slug becomes the stable public identifier used by EP-4 (discovery) and EP-5 (purchase).
- **Lifecycle transition:** Draft → Published is the only valid publish transition. Published, Closed, and Cancelled events cannot be published again.
- **EVT-EventPublished:** An integration-scope domain event emitted when publishing succeeds. Downbound consumers (Reporting/Discovery) use this to make the event visible.
- **Ownership enforcement:** Only the user holding the Owner role (F-1.5) for the specific event may publish it. This is an application-layer authorization check, not a domain invariant.

## 4. API Contract

**Endpoint:** `POST /api/events/{eventId}/publish`

**Auth:** Cookie session required. The handler verifies the caller holds the Owner role for the event.

**Success (200):** Returns the updated event representation with status `Published` and the generated public slug/URL.

**Error responses:**

| Status | Code | Condition |
|--------|------|-----------|
| 401 | `UNAUTHORIZED` | No active session |
| 403 | `FORBIDDEN` | Caller does not hold the Owner role for this event |
| 404 | `NOT_FOUND` | Event does not exist |
| 422 | `EVENT_NOT_PUBLISHABLE` | Missing required details or no ticket type — response includes a list of specific unmet requirements |
| 422 | `EVENT_ALREADY_PUBLISHED` | Event is already Published |
| 422 | `INVALID_EVENT_STATUS` | Event is Closed or Cancelled |

## 5. Data & Storage Impact

- **PostgreSQL:** The event's status field transitions from `Draft` to `Published`. The slug field is populated on publish. Both are persisted in the `events` table (`app` schema). The `row_version` optimistic concurrency token is incremented.
- **Redis:** No direct cache impact on publish. The event becomes queryable through the public discovery path (EP-4), which may cache independently.
- **MinIO:** No change — cover image (F-2.2) is already stored by reference.
- **RabbitMQ:** An `EventPublished` integration message is emitted for downstream consumers (Reporting/Discovery projections).

## 6. Real-Time & Consistency

- **Strong consistency:** The publish action (status change + slug generation) occurs within a single transaction on the Event aggregate. Optimistic concurrency prevents conflicting writes.
- **Integration event:** `EVT-EventPublished` is emitted after successful commit. Downstream consumers (reporting projections, discovery indexes) update eventually — the event may take a brief moment to appear in listings or search results.
- **N/A for SignalR (MVP):** Real-time push for publish events is not required for this feature slice. It becomes relevant in EP-11.

## 7. Security & Privacy

- **Session auth required:** The caller must be signed in. No guest or anonymous publish.
- **Owner-only authorization:** Checked in the Application handler via `ICurrentUserAccessor` — not just at the API endpoint level.
- **No sensitive data exposed:** The publish response returns event metadata and slug only. No payment or personal data is involved.
- **Slug generation safety:** Slugs are URL-safe, non-guessable-enough for public use, and collision-checked against existing published events.

## 8. Edge Cases

**EC-01:** Two organizers try to publish events that would generate the same slug simultaneously. The system detects the collision and generates a unique variant (e.g., appends a short suffix) so both succeed without conflict.

**EC-02:** An organizer adds a ticket type, publishes, then deletes the ticket type. The event remains Published but is flagged as invalid — the system surfaces a warning to the organizer that the event has no ticket types and is not fully functional. The event is not automatically reverted to Draft.

**EC-03:** The event has required details but the ticket type has zero capacity. The event is still publishable — zero-capacity ticket types are a valid configuration (they show as sold out immediately). This is the organizer's choice.

**EC-04:** The event's row_version has changed between the owner loading the page and clicking Publish (concurrent edit by another session). The system returns a conflict error (409) and the owner must refresh.

**EC-05:** An event was previously Published, then Closed, and the owner wants to re-open it. Re-publishing is not allowed — Close is a one-way transition for this feature. Re-opening may be a future enhancement.

## 9. Dependencies & Risks

**Dependencies:**
- F-2.1 (Create a draft event) — must exist before publishing
- F-3.1 (Define a ticket type) — at least one ticket type is required
- F-1.5 / F-1.6 (Roles) — Owner role must be assignable and checkable

**Risks:**
- Slug generation performance at scale is low-risk for MVP (small event volume). If needed, slugs can be pre-computed or use a simpler scheme.
- The "at least one ticket type" gate couples publishing to the ticketing epic (EP-3). This is intentional — publishing without something to sell has no value in the MVP.

## 10. Assumptions

- The slug is generated from the event title (sanitized, URL-safe) with collision resolution. The exact slug algorithm is an implementation detail.
- A Draft event that has been published and then Closed cannot be re-published. Closing is a terminal state for the publish lifecycle.
- If a Published event ends up with zero ticket types (all deleted), the system flags it as invalid rather than auto-reverting to Draft.
- The publish action does not trigger notifications to anyone (no email blast on publish). Notifications are a later feature (F-9.5).
- The stable public URL format is `/events/{slug}` — the slug is the permanent identifier, not the internal event ID.

## 11. Out of Scope

- **Unpublish / revert to Draft:** Once published, an event cannot go back to Draft. Close (F-2.5) or Cancel (F-2.5) are the available transitions.
- **Scheduled publish:** Publishing happens immediately on the owner's action. Auto-scheduling publish for a future time is not included.
- **Publish validation beyond core requirements:** No checks for description quality, image presence, or pricing completeness beyond "has at least one ticket type."
- **Bulk publish:** One event at a time.
- **Discovery page integration (EP-4):** The public listing and search consume the published state but are separate features.
